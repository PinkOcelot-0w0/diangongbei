# 项目完善设计说明

## 目标

基于论文《B题_嵌入式社区养老服务站建设与优化_设计论文.docx》第七节的项目核查建议，补齐项目的复现说明、测试、追踪文档和关键模型约束，使当前仓库能够说明“如何运行、结果从哪里来、哪些约束被检查、哪些风险仍需进一步建模”。

## 范围

本次采用方案 A：保持论文当前主体结果可复现，同时修复工程层面的明显缺口。

- 问题一：从附件 1 的“转移概率”工作表读取年度转移概率，避免源码硬编码。
- 问题二：默认完整枚举 10 个候选站点，保留软容量模型，并额外输出容量核查表；不直接改成硬容量模型，以免偏离论文当前表格结论。
- 问题四：在统一 markup 定价搜索中增加站点利润非负约束，并在不可行时输出约束说明。
- 工程文档：新增 README、AGENTS 和 docs 知识库入口，说明运行顺序、模型边界、输出文件、测试方式和已知风险。
- 测试：新增基于 `unittest` 的回归测试，覆盖 Excel 读取、距离矩阵、候选点枚举、容量核查和 q4 利润约束。

## 非目标

- 不重写四个题解脚本为包结构。
- 不修改论文 DOCX。
- 不把 q2 改成默认硬容量最优解。
- 不新增第三方依赖。

## 架构

项目继续保留四个独立脚本：`q1_solution.py` 生成第 5 年人口和需求，`q2_solution.py` 基于 q1 输出做选址规模优化，`q3_solution.py` 基于 q2 输出做分站分服务定价，`q4_solution.py` 做统一 markup 灵敏度方案。新增的文档和测试围绕这些脚本建立追踪关系，不改变原始数据文件位置。

## 数据流

```text
附件1/附件2
  -> q1_solution.py
  -> q1_output/*.csv
  -> q2_solution.py + 附件3/附件4
  -> q2_output/*.csv
  -> q3_solution.py + 附件2/附件3
  -> q3_output/*.csv
  -> q4_solution.py
  -> q4_output/*.csv
```

## 错误处理

- Excel 工作表、单元格或概率值缺失时抛出带中文说明的 `ValueError`。
- q4 若没有满足利润率上限和利润非负的 markup 组合，输出空结果文件和 `infeasibility_report.txt`。
- q2 软容量过载不作为失败，但通过 `capacity_check.csv` 明确标记。

## 测试策略

使用标准库 `unittest`，避免新增依赖。测试通过命令：

```bash
python -m unittest discover -s tests -v
```

核心验证包括：

- q1 读取到附件中的 0.045 和 0.1。
- 距离矩阵为 10x10 且对称。
- q2 默认候选站点数量等于距离矩阵全部站点数量。
- q2 容量核查表包含过载标记。
- q4 不接受负利润站点作为可行解。
